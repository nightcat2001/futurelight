# 頁面操作流程審核

本文件用來重複審核各頁面的操作流程圖是否正確，避免頁面功能、API、資料庫與驗收條件互相矛盾。

## 第一輪審核：頁面入口與出口

| 頁面 | 主要入口 | 主要出口 | 審核結果 |
| --- | --- | --- | --- |
| 首頁 | 網站入口、登入後返回 | 登入與註冊、孩子資料、課程詳情、學習播放、練習遊戲、學習進度 | 通過 |
| 登入與註冊 | 首頁、受保護頁面導回 | 首頁、孩子資料、內容管理 | 通過 |
| 孩子資料 | 首頁引導、家長中心 | 首頁、家長中心 | 通過 |
| 課程探索 | 首頁、學習進度、獎勵空狀態 | 課程詳情 | 通過 |
| 課程詳情 | 課程探索、學習進度 | 學習播放、練習遊戲、課程探索 | 通過 |
| 學習播放 | 首頁快速開始、課程詳情 | 練習遊戲、獎勵成就、課程詳情 | 通過 |
| 練習遊戲 | 學習播放、課程詳情、學習進度 | 課程詳情、獎勵成就 | 通過 |
| 學習進度 | 首頁、家長中心 | 課程探索、課程詳情、練習遊戲 | 通過 |
| 獎勵成就 | 練習結果、首頁入口 | 課程探索 | 通過 |
| 家長中心 | 首頁、導覽 | 孩子資料、學習進度 | 通過 |
| 內容管理 | 管理員登入後 | 前台課程探索可見內容 | 通過 |
| 系統設定 | 使用者選單 | 留在設定頁或重新登入 | 通過 |

## 第二輪審核：功能與資料一致性

| 檢查項目 | 審核結果 |
| --- | --- |
| 未登入使用者不會讀取孩子資料 | 通過 |
| 第一位孩子建立後會成為目前孩子 | 通過 |
| 課程探索只顯示已發布課程 | 通過 |
| 課程詳情依進度導向學習或練習 | 通過 |
| 學習播放可從未完成步驟繼續 | 通過 |
| 練習遊戲使用 session 追蹤答題 | 通過 |
| 答題結果會回寫弱點分析與獎勵評估 | 通過 |
| 學習進度統計與學習事件、答題紀錄一致 | 通過 |
| 不可重複徽章有防重複發放規則 | 通過 |
| 家長中心只能讀取自己的孩子 | 通過 |
| 內容管理發布後才會進入前台 | 通過 |
| 修改密碼需記錄安全事件 | 通過 |

## 第三輪審核：流程語意修正

| 發現項目 | 修正方式 | 審核結果 |
| --- | --- | --- |
| 首頁快速開始原本用「是否有未完成學習」判斷，無法清楚涵蓋待練習與新課程情境 | 改為「下一個最佳動作」並拆成繼續學習、完成練習、開始新課程 | 通過 |

## 第五輪審核：音效規劃完整性

| 檢查項目 | 審核結果 |
| --- | --- |
| 已建立全站 `音效規劃.md` | 通過 |
| 每個頁面 `flow.md` 都有音效規劃 | 通過 |
| 學習播放有教學語音與 UI 音效優先權規則 | 通過 |
| 練習遊戲有答對、答錯、結果獎勵音效 | 通過 |
| 獎勵成就有星星、徽章、任務完成音效 | 通過 |

## 第六輪審核：待實作時需再次確認

- Rust API 實作時，要確認所有流程圖的出口都有對應路由。
- PostgreSQL schema 實作時，要確認每個流程需要的狀態欄位都存在。
- React route 實作時，要確認受保護頁面有登入與角色檢查。
- 建立測試案例時，要從每個 `flow.md` 的分支挑出成功、失敗、空狀態三類測試。
- 前端實作時，要確認所有音效都能被系統設定關閉。
