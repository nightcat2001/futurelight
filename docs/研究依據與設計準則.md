# 研究依據與設計準則

本文件整理 FutureLight 在介面、互動、動畫、音效與兒童學習體驗上的外部參考來源。目的不是照抄某個設計系統，而是把有研究或成熟實務基礎的原則轉成專案可執行規範。

## 對 FutureLight 的設計結論

### 1. 每個操作都要有可見回饋

依據 NN/g「系統狀態可見」與兒童互動研究，孩子點擊後不能只改資料，畫面要立刻告訴孩子「我有收到」。

落地規則：

- 主要按鈕點擊後需有 pressed/loading/success/error 狀態。
- 學習步驟完成後需有視覺回饋與短音效。
- 練習答題後需立即顯示答對或答錯，但答錯不能羞辱孩子。
- 儲存、發布、上傳等後台操作需顯示成功或失敗訊息。

### 2. 導航要靠辨識，不靠記憶

孩子不應該記住「下一步要去哪」。介面要直接給出下一個合理按鈕。

落地規則：

- 首頁提供 `快速開始`，不要求孩子自己找未完成單元。
- 課程詳情依狀態顯示 `開始學習`、`繼續學習`、`練習`、`複習`。
- 學習播放完成後自動推薦下一步：練習、獎勵或返回課程。
- 家長中心要能從孩子摘要直接進入進度與管理資料。

### 3. 兒童介面要降低文字負擔

兒童學習產品不適合把所有說明都塞成文字。研究指出兒童互動會同時依賴情緒、感官、動作與語言線索。

落地規則：

- 學習與練習頁優先用圖片、語音、短句與狀態圖示。
- 長文字留給家長中心、內容管理與系統設定。
- 錯誤訊息要短，並提供下一步。
- 重要操作不能只靠顏色，要搭配文字、圖示或音效。

### 4. 音效是回饋，不是噪音

Material sound 指出聲音應提供資訊並讓使用者安心。兒童語言學習中，教學語音優先於 UI 音效。

落地規則：

- 教學語音、聽力題音檔優先權最高。
- UI 點擊音效要短、低音量、可關閉。
- 答錯音效使用柔和提示，不使用懲罰或失敗感強的聲音。
- 獎勵音效只在新獎勵發生時播放，不在每次進頁重播。
- 系統設定必須提供音效開關、語音音量與 UI 音效音量。

### 5. 動畫要幫助理解空間關係

Material motion 建議動畫既要快，也要讓使用者理解轉場。WCAG 則要求避免不可控的持續動態。

落地規則：

- 一般頁面換頁使用 150-220ms。
- 學習流程前進可用 220ms 左右的方向性轉場。
- Tab 切換使用 120ms 淡入，避免大幅滑動。
- 獎勵動畫可稍長，但應控制在 300ms 左右。
- 若使用者啟用 reduced motion，動畫需縮短或取消。
- 自動動畫不可持續超過 5 秒且無控制方式。

### 6. 兒童流程要有鷹架支持

兒童 App 研究提到 clear goals、timely feedback、scaffolding。FutureLight 需要讓孩子知道現在在哪、下一步做什麼、完成後得到什麼。

落地規則：

- 學習播放頁顯示 `步驟 2 / 5`。
- 練習遊戲頁顯示 `題目 3 / 10`。
- 任務顯示進度，例如 `今天完成 1 個單元 0 / 1`。
- 完成後不要只跳頁，要先顯示結果或獎勵。
- 弱點建議要能直接導向可執行的練習。

### 7. 設計系統參考方向

FutureLight 可以參考 Paragon 與 Instructure UI 的方向，但不必一開始導入完整套件。

建議：

- React 元件先建立自己的輕量 design tokens：color、spacing、radius、motion、sound。
- 可參考 Paragon 的教育產品元件組織方式與 design tokens。
- 可參考 Instructure UI 對鍵盤、螢幕閱讀器、RTL 的要求。
- 若未來要導入元件庫，優先選擇 accessibility-first、React-friendly、可 theme 的系統。

## 對現有文件的修正要求

- `互動導向規劃.md` 的動畫時間需符合 Material motion 的範圍。
- `音效規劃.md` 需保留音效可關閉與教學語音優先規則。
- 各頁 `flow.md` 的按鈕操作需符合「可見回饋」與「辨識優於記憶」。
- `流程審核.md` 需加入研究依據審核，避免後續改動失去根據。

## 後續實作驗證

前端完成後應做以下驗證：

- 使用鍵盤完成主要流程。
- 關閉音效後完成主要流程。
- 啟用 reduced motion 後完成主要流程。
- 孩子流程不需要閱讀長段說明即可完成。
- 每個主要按鈕點擊後 100ms 內有視覺回饋。
- 每個 API 操作都有 loading、success、error 三種狀態。
