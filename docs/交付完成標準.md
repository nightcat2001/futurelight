# 交付完成標準

Created: 2026-06-01 10:10:18 +08:00

FutureLight 的完成定義是「真實功能可驗證」，不是畫面看起來存在。任何工作若只完成 mock、固定 JSON、假按鈕、假成功訊息、未落地素材或未接資料流，都不得標記為完成。

## 1. 全域 No-Mock Checklist

每次交付前必須逐項確認：

- [ ] 沒有把 mock、固定 JSON、硬編成功狀態標記為完成。
- [ ] 未完成操作在 UI 上明確 disabled、coming soon 或 blocked。
- [ ] 主要按鈕有真實 route、真實 API mutation 或清楚的 blocked state。
- [ ] 所有 loading、success、error 狀態都是真實流程結果。
- [ ] 驗收證據已寫回 Todo 或交付說明。

## 2. Frontend 完成標準

- React route 必須是真實 URL，支援 browser back 與深連結。
- 頁面資料必須來自 API、DB-backed seed/import，或明確標為尚未完成。
- 表單提交必須呼叫真實 API；不可只顯示成功 toast。
- 受保護頁面必須有登入/角色檢查。
- 未完成功能不可藏在可點擊假按鈕後。
- 兒童區不得出現購買入口、外部連結、廣告或未審核 AI 內容。

## 3. Rust API 完成標準

- Handler 必須連到 service/repository/domain model。
- API 不可只回固定 JSON 當完成。
- 寫入型 API 必須有 validation、error model 與 audit/logging 策略。
- 兒童資料 API 必須檢查 parent-child ownership。
- 若 DB 尚未存在，該 API 只能標為 blocked，不可標成完成。

## 4. PostgreSQL 完成標準

- Schema 變更必須透過 migration。
- 測試資料需是明確 seed/import，不是前端硬編。
- 可從空 DB 重建 local/staging schema。
- 刪除、同意、匯出、學習紀錄等敏感流程需有 audit_logs。
- migration 與 API 欄位必須與 docs 對齊。

## 5. Assets / Audio / Video 完成標準

- 圖片、音效、影片必須有實際檔案。
- 所有素材必須登記在 `assets/asset_manifest.json` 或後續 DB-backed asset table。
- AI 素材需記錄 prompt、模型、日期、用途、授權狀態、審核者。
- 音效不可只列 sound ID；必須有檔案、播放規則、音量控制與關閉能力。
- Remotion / HyperFrames 必須有可 render 或可預覽 composition。
- Seedance 輸出必須有影片檔，或留下 RunComfy request/blocker，不可只留想法。

## 6. Browser / QA 完成標準

- 每次交付必跑命令以 `docs/交付驗證命令清單.md` 為準。
- 預設總驗證命令是 `powershell.exe -NoProfile -ExecutionPolicy Bypass -File .\tools\verify-delivery.ps1`。
- 需要 UI 驗證的工作，必須有真實點擊流程、截圖或 smoke test。
- 僅頁面可開啟不代表完成；主要 CTA 也必須可操作或明確 blocked。
- 桌面與手機 viewport 不得有文字重疊、按鈕溢出或主要內容被遮擋。
- 若未啟動服務，交付說明需明確寫出未啟動原因與已跑的替代驗證。

## 7. Port / Service 完成標準

- 啟動服務前必須讀 `workspace-port-guard` 並跑 port policy check。
- FutureLight 僅使用 frontend `37173`、API `37200`、PostgreSQL `37432`。
- Vite 必須使用 strict port。
- Docker host port 必須與 registry 一致。
- 不可使用 Vite/Next/Cargo/Docker 自動 fallback port。

## 8. 交付說明必填

每次完成工作，交付說明需包含：

- 完成的 backlog ID。
- 修改的真實檔案。
- 驗證命令與結果。
- 未啟動服務或未做瀏覽器驗證的原因。
- 下一個依序 backlog ID。
